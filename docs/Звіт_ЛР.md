# Звіт про виконання лабораторної роботи

## Постановка завдання

Реалізувати підтримку текстового опису багатопотокової програми у вигляді блок‑схем та її симуляцію з перебором можливих міжпотокових розкладів.

Мова опису потоку повинна підтримувати:

- **спільні змінні** (секція `VARS:`), доступні всім потокам;
- **команди**:
  - `V = C` — присвоєння константи,
  - `V1 = V2` — присвоєння значення змінної,
  - `INPUT V` — зчитування зі стандартного вводу,
  - `PRINT V` — виведення значення спільної змінної `V` на стандартний вивід;
- **умови в блоках розгалуження**:
  - `V == C` — “значення V рівне значенню C”,
  - `V < C` — “значення V менше значення C”.

Для команд після `->` задається **один** наступний вузол `nextId` (або `-1` для завершення потоку).
Для умов після `->` задаються **два** вузли через кому: `trueNextId, falseNextId`.

Окремо описується тест:

- `TEST:`
- `INPUT:` — список цілих (може бути порожнім)
- `EXPECTED:` — очікуваний вихід (список цілих)
- `K:` — обмеження на кількість кроків симуляції

## Вхідні дані та формат

### Загальна структура файлу прикладу

```text
VARS: x, y

THREAD T1:
  1: INPUT x -> 2
  2: x < 0 -> 3, 4
  3: PRINT x -> -1
  4: PRINT x -> -1

TEST:
INPUT: 0
EXPECTED: 0
K: 10
```

### Приклади, що додаються до проєкту

- `samples/correct_conditions.txt` — базовий приклад з `PRINT`, `V==C`, `V<C`.
- `samples/myTest_with_conditions.txt` — приклад на основі `samples/myTest.txt`, але з умовами `V==C` та `V<C`.
- `docs/Команди_та_умови.md` — коротка документація синтаксису команд і умов.

## Архітектура та компоненти

Проєкт побудований як Java‑застосунок (Maven) з розділенням на логічні шари.

### 1) Модель (package `org.example.model`)

- **`ProgramSpecification`**: перелік спільних змінних + список блок‑схем (по одній на потік).
- **`Flowchart`**: граф одного потоку:
  - `statementNodes`: вузли‑інструкції (одна наступна вершина),
  - `conditionNodes`: вузли‑умови (дві гілки: true/false).
- **`StatementNode` / `StatementType`**: опис інструкції (`ASSIGN_CONST`, `ASSIGN_VAR_VAR`, `INPUT`, `PRINT`).
- **`ConditionNode` / `ConditionOp`**: опис умови:
  - `EQ` для `==`,
  - `LT` для `<`.

### 2) Парсер (package `org.example.parser`)

- **`TextFlowchartParser`**:
  - читає текстовий формат (`VARS`, `THREAD`, кроки, `TEST`);
  - для рядка кроку визначає, чи це **умова** (`V==C` або `V<C`) чи **інструкція**;
  - для умов очікує два `nextId` через кому (`trueNextId, falseNextId`) і створює `ConditionNode`;
  - для інструкцій парсить `V=C`, `V1=V2`, `INPUT V`, `PRINT V` і створює `StatementNode`.

### 3) Симуляція (package `org.example.simulation`)

- **`NaiveProgramSimulator`**:
  - наївно перебирає **всі** можливі міжпотокові розклади виконання (DFS) в межах ліміту `maxSteps`;
  - стан містить:
    - `pcs[]` — “program counter” для кожного потоку (id поточного вузла),
    - `vars[]` — значення спільних змінних,
    - `inputPos` — позицію у вхідних даних,
    - `output` — накопичений вивід.
  - обробляє:
    - `PRINT V` — додає поточне значення `V` до `output`,
    - `V==C` та `V<C` — обирає `trueNextId` або `falseNextId`.
  - порівнює `output` з `EXPECTED` і формує `ExecutionResult`.

### 4) Графічний інтерфейс (package `org.example.gui`)

- **`FlowchartEditorFrame`**:
  - надає вікно редактора для введення текстового опису;
  - викликає `TextFlowchartParser` для побудови `ProgramSpecification`;
  - запускає симуляцію та відображає результат перевірки.

## Опис реалізації (що саме було зроблено)

У межах лабораторної роботи забезпечено коректну підтримку умов у розгалуженнях:

- **`V == C`** (операція `EQ`);
- **`V < C`** (операція `LT`).

Ключова частина реалізації — **парсер**:

- додано розпізнавання конструкцій `V==C` та `V<C`;
- для таких рядків підтримано формат переходів після `->` як `trueNextId, falseNextId`;
- створення `ConditionNode` та додавання його до `Flowchart`.

Симулятор уже містив виконання `EQ` і `LT`, тому після розширення парсера нові умови почали працювати “наскрізно”: **файл → парсер → модель → симулятор → результат**.

## Інструкція запуску та перевірки

### Збірка

У корені проєкту:

```bash
mvn compile
```

### Перевірка на прикладах

1. Відкрити один із файлів прикладів у редакторі застосунку (або скопіювати вміст у GUI):
   - `samples/correct_conditions.txt`
   - `samples/myTest_with_conditions.txt`
2. Запустити симуляцію.
3. Переконатися, що:
   - `PRINT V` додає значення змінної `V` у вихід,
   - умови `V==C` та `V<C` обирають правильну гілку виконання.

## Екранні знімки (за потреби)

Вставити у цей розділ скріншоти:

- **Скріншот 1:** Вікно редактора з прикладом `myTest_with_conditions.txt`.
- **Скріншот 2:** Результат симуляції / перевірки виходу (вивід та ознака збігу з `EXPECTED`).

## Висновки

Реалізовано підтримку команд `PRINT V` та умовних операторів `V==C` і `V<C` у текстовому описі блок‑схем. Парсер коректно будує модель програми з умовними вузлами, а симулятор виконує гілкування та формує вихід для перевірки коректності відносно очікуваного результату.
